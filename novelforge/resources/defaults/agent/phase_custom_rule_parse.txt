# 结构化任务
请将用户原始输入结构化为以下四字段 JSON 对象：

1. **title**：简短标题（≤20 字），概括该设定的核心要求。如用户输入"主角不能在白天使用能力"则 title 可为"主角白天禁用能力"。
2. **requirement**：续写生成时的硬约束描述。要求具体、可执行、可验证，注入续写提示词后 LLM 能直接遵守。如"续写中主角在白天（日出到日落）场景下不得使用任何超自然能力，必须依靠智谋、装备或他人协助应对冲突"。
3. **audit_criteria**：审计检查向。逐条列出审计员需检查的项目，每条应可独立判定通过/不通过。如"① 检查正文是否存在白天场景下主角使用超自然能力的描写；② 检查主角在白天场景应对冲突时是否依赖非能力手段；③ 检查是否有交代白天禁用能力的设定依据"。
4. **severity**：严重程度，取值 `critical`（一票否决：违反时审计 passed=false）或 `major`（扣分但不否决：违反时该维度低分但整体仍可 passed=true）。默认 `critical`，仅当用户输入明确为软性建议或风格偏好时取 `major`。

# 结构化原则
- **具体化**：将用户模糊描述转化为具体可检查的要求，避免"适当""合理"等模糊词
- **可验证**：requirement 与 audit_criteria 必须可被审计员客观判定，不依赖主观感受
- **保变量**：用户输入中的 {{book}}/{{protagonist}}/{{char}} 等变量占位符在结构化时保留原样，续写与审计时由宏引擎替换
- **结合世界观**：结构化时参考世界观底层，确保 requirement 不与底层元规则冲突；如冲突则在 audit_criteria 中增加"冲突标注"检查项
- **结合上下文**：结构化时引用上下文条目中的具体人物/地点/事件，避免泛泛而谈
- **逐条化**：audit_criteria 用 ①②③ 序号分条，每条独立可判定

# 输出格式
请输出严格的 JSON 对象，包含以下字段：
- title：字符串，简短标题（≤20 字）
- requirement：字符串，续写生成硬约束描述（具体、可执行、可验证）
- audit_criteria：字符串，审计检查向（逐条用 ①②③ 序号分条，每条独立可判定）
- severity：字符串，严重程度，取值 `critical` 或 `major`

注意：
- 只输出 JSON 对象，不要包含任何解释性文字
- 不要使用 markdown 代码块标记
- 所有字段必须存在
- 确保 JSON 语法严格合法，可直接被解析
