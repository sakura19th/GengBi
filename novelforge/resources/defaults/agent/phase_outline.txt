# 规划任务
请综合参考前文快照（人物状态、情节线、伏笔、承诺、风格等）、前文章节文本以及用户续写指令，规划 3-7 个场景的续写大纲。每个场景应服务于整体叙事推进，兼顾冲突升级、伏笔回收与风格一致性。

# 输出格式
请输出严格的 JSON 对象，包含以下字段：
- continuation_goals：字符串，本次续写的总体目标（要达成什么叙事效果、推进哪些情节线）
- foreshadowing_plan：字符串，本次续写中伏笔的埋设与回收计划
- scenes：数组，场景大纲，每项含：
  - purpose：字符串，该场景的叙事目的（如"揭示人物动机"、"升级冲突"、"埋设伏笔"等）
  - pov：字符串，该场景的叙事视角（如"主角第三人称限知"、"反派视角"等）
  - scene_type：字符串，场景类型（如"对话/动作/心理/过渡/高潮"等）
  - goal：字符串，场景中视点人物的目标
  - conflict：字符串，场景中的冲突与阻碍
  - outcome：字符串，场景的结局与结果（成功/失败/妥协/反转）
  - value_shift：字符串，场景中人物价值观或关系的转变（从+到-、从-到+等）
  - foreshadowing：字符串，该场景涉及的伏笔埋设或回收说明
  - exit_hook：字符串，场景结尾的钩子（吸引读者继续阅读的悬念）

# 规划原则
- 一致性优先：场景规划必须与前文快照中的人物状态、情节线、世界状态保持一致
- 冲突驱动：每个场景都应有明确的冲突与目标，避免无张力过渡
- 节奏把控：3-7 个场景应形成合理的张力曲线（起-承-转-合）
- 伏笔回收：优先回收快照中 status 为 planted 的伏笔，同时可埋设新伏笔
- 兑现承诺：推进 unresolved_promises 中的叙事承诺
- 风格统一：场景类型与视角选择应与前文 style_profile 保持一致
- 用户意图：若用户指令非空，应在规划中体现用户意图

注意：
- 只输出 JSON 对象，不要包含任何解释性文字
- 不要使用 markdown 代码块标记
- 所有字段必须存在，无内容时使用空字符串
- scenes 数组长度必须在 3-7 之间
- 确保 JSON 语法严格合法，可直接被解析
