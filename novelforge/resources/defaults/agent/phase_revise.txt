# 修订指导任务
请基于评审报告中指出的问题以及续写大纲的规划目标，产出结构化的修订指导。注意：你的职责是给出修订方向与策略，而非直接改写文本。修订指导应具体、可操作，使后续修订环节能据此高效完成改写。

# 输出格式
请输出严格的 JSON 对象，包含以下字段：
- revision_strategy：字符串，总体修订策略（概括本次修订的核心方向、优先级与整体思路）
- key_changes：数组，关键修订项，每项含：
  - issue_ref：字符串，对应评审报告中问题的引用标识（如 category+location 或问题描述摘要）
  - revision_action：字符串，具体修订动作（如何修改、改向什么方向，而非直接给出改写文本）
  - target_section：字符串，修订目标位置（场景编号、段落或文本片段引用）
- preserve_elements：字符串，需要保留的优质元素（续写文本中表现良好、不应在修订中破坏的部分，如某段精彩对话、某处有效伏笔等）

# 修订指导原则
- 针对性：每项 key_changes 应对应评审报告中的具体问题
- 可操作：revision_action 必须给出明确的改写方向（如"将第2场景的对话压缩为半页"、"在第3场景增加主角内心独白揭示动机"），而非泛泛而谈
- 一致性：修订方向需与前 10 章前文设定（人物/地点/事件/风格）保持一致，避免引入与前文矛盾的设定
- 节奏匹配：修订后的场景密度需与 pacing_speed 匹配（slow=同一场景1-2章，单章场景1-3个/medium=一个场景半章，单章场景3-5个/fast=快速，单章场景5-8个）
- 不越界：只给方向与策略，不直接产出改写后的文本
- 平衡性：在修复问题的同时，明确指出需要保留的优质元素，避免修订破坏已有亮点
- 大纲对齐：修订方向应有助于更好地落实大纲规划，而非偏离

注意：
- 只输出 JSON 对象，不要包含任何解释性文字
- 不要使用 markdown 代码块标记
- 所有字段必须存在，无内容时使用空字符串或空数组
- key_changes 应覆盖评审报告中所有 critical 与 major 级别问题
- 确保 JSON 语法严格合法，可直接被解析
